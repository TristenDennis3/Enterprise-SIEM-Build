OVERVIEW

Wazuh Manager 4.14.5 was deployed on an Ubuntu Server 24.04 VM in the lab. Because the lab subnet has no internet access, the online installation method was not viable. The offline procedure was used instead: all installation artifacts were prepared on an internet-connected staging VM, transferred into the air-gapped environment via an ISO pipeline, and installed locally on the target. This document covers the full process from artifact preparation through service verification.

SECTION 1 — STAGING THE OFFLINE DOWNLOADER

A temporary internet-connected Ubuntu Server VM was created in VirtualBox on the management workstation to download and prepare the offline artifacts (NAT networking, 2 GB RAM, 2 cores, 20 GB disk).

1.1 Create working directory

mkdir -p ~/wazuh-offline

cd ~/wazuh-offline

1.2 Download the installation assistant

curl -sO https://packages.wazuh.com/4.14/wazuh-install.sh

chmod 744 wazuh-install.sh

1.3 Download and edit the configuration template

curl -sO https://packages.wazuh.com/4.14/config.yml

nano config.yml

All three node IPs (indexer, server, dashboard) were set to the single target Wazuh manager for an all-in-one deployment.

1.4 Download the offline package bundle

sudo bash wazuh-install.sh -dw deb -da amd64

Note: The architecture flag must match the package format — DEB requires amd64 (RPM would use x86_64). A mismatch aborts the installer. This produces wazuh-offline.tar.gz (~1.6 GB).

1.5 Generate cluster certificates

sudo bash wazuh-install.sh -g

Produces wazuh-install-files.tar (~10 KB), containing the root CA, indexer, filebeat, dashboard, and admin certificates plus the cluster key.

1.6 Verify the three artifacts

ls -la

Confirms wazuh-install.sh, wazuh-offline.tar.gz, and wazuh-install-files.tar are all present.

SECTION 2 — TRANSFERRING ARTIFACTS TO THE TARGET VM

The target VM has no internet and no inbound SSH, so a multi-stage offline pipeline (shared folder → ISO → Proxmox) was used.

2.1 Downloader → management workstation (VirtualBox shared folder)

sudo apt update

sudo apt install -y virtualbox-guest-utils

sudo usermod -aG vboxsf $USER

sudo reboot

After reboot, the shared folder was mounted manually and the artifacts copied to it (sudo required for the two root-owned files):

sudo mkdir -p /mnt/share

sudo mount -t vboxsf wazuhfiles /mnt/share

cp wazuh-install.sh /mnt/share/

sudo cp wazuh-install-files.tar /mnt/share/

sudo cp wazuh-offline.tar.gz /mnt/share/

2.2 Package artifacts into an ISO (AnyBurn)

On the workstation, AnyBurn was used to build wazuh-files.iso from the three artifacts: Create image file from files/folders → add the files → output as ISO → save as wazuh-files.iso. Proxmox accepts ISO 9660 images as virtual CD-ROMs, which is why this format was used.

2.3 Upload and attach the ISO in Proxmox

The ISO was uploaded to the Proxmox node's local storage (ISO Images tab → Upload), then attached to the target Wazuh VM as a virtual CD-ROM via the VM's Hardware tab.

2.4 Mount and copy artifacts on the target VM

sudo mount /dev/sr0 /mnt

ls /mnt

cp /mnt/wazuh-install.sh ~

cp /mnt/wazuh-install-files.tar ~

cp /mnt/wazuh-offline.tar.gz ~

Note: A read-only warning on mount is expected — ISO 9660 is a read-only filesystem.

SECTION 3 — RESOLVING DEPENDENCIES

The offline installer validates that several Ubuntu base packages are present before it will proceed. On the first install attempts it reported missing dependencies; a few base packages had to be resolved manually before the installer would run. Each was downloaded on the internet-connected workstation and transferred to the target through the same ISO pipeline used for the Wazuh artifacts.

3.1 Verify dependencies

The required base packages were checked with dpkg to identify which were missing:

dpkg -l | grep -E "debconf|adduser|procps|apt-transport-https|gnupg|debhelper|libcap2-bin"

3.2 Acquire and transfer the missing packages

The matching .deb packages for Ubuntu 24.04 were downloaded from the Ubuntu archive (archive.ubuntu.com/ubuntu/pool/) on the workstation — each matched to the target's installed apt version — and moved to the target via the ISO pipeline.

3.3 Install the dependencies

sudo dpkg -i /mnt/*.deb

Note: Each missing dependency was installed until the installer's dependency check passed. apt-transport-https was among those required.

3.4 Import the Wazuh GPG key

The Wazuh GPG key was extracted from the offline bundle and imported so the installer could verify package signatures:

tar -xzf wazuh-offline.tar.gz

sudo gpg --no-default-keyring --keyring /usr/share/keyrings/wazuh.gpg --import wazuh-offline/wazuh-files/GPG-KEY-WAZUH

sudo gpg --no-default-keyring --keyring /usr/share/keyrings/wazuh.gpg --list-keys

SECTION 4 — WAZUH INSTALLATION

4.1 Confirm working directory

cd ~

ls -la

All three artifacts (wazuh-install.sh, wazuh-install-files.tar, wazuh-offline.tar.gz) must be in the working directory.

4.2 Run the offline all-in-one installation

sudo bash wazuh-install.sh -a -o

The -a flag performs an all-in-one deployment (indexer, manager, dashboard on one host); -o (--offline-installation) installs from the local bundle without internet access. The admin credentials are printed at the end of the run and must be captured immediately — they are not redisplayed.

4.3 Confirm the install completed

sudo systemctl status wazuh-manager

sudo tail -50 /var/log/wazuh-install.log

SECTION 5 — NETWORK CONFIGURATION

The host was assigned a static address on the SIEM VLAN via Netplan:

sudo nano /etc/netplan/50-cloud-init.yaml

sudo netplan apply

ip -4 a show ens18

ip route

Static address set to 10.0.20.20/24, gateway 10.0.20.1.

SECTION 6 — VERIFICATION

sudo systemctl status wazuh-manager

sudo systemctl status wazuh-indexer

sudo systemctl status wazuh-dashboard

All three services should report active (running). The dashboard was then reached at https://10.0.20.20; the self-signed certificate warning was accepted (certificates were generated locally), and login succeeded with the admin credentials captured during installation.

LESSONS LEARNED

Topic

Detail

Architecture flag

DEB packages require amd64; using x86_64 (the RPM convention) aborts the installer.

Missing dependencies

A few base packages were missing on the Ubuntu 24.04 image and had to be installed manually; each had to match the target's apt version.

Misleading error

Failed runs reported “GPG key not found,” but the real cause was the missing apt-transport-https.

Offline flag

-o (--offline-installation) is required; -a alone attempts internet-dependent steps.

Working directory

All three artifacts must be in the working directory for the installer to run.
