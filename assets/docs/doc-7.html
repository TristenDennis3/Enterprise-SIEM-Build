1. OVERVIEW

I ran an attack chain starting as a standard domain user and ending with control of another machine and every credential in the domain. I ran this attack chain to then view Windows Event Viewer logs and sift through Security Onion and Wazuh events, to help me practice identifying the IDs related to these attacks.

2. THE ATTACK CHAIN

Stage 1 — Kerberoasting

Authenticated as a standard user (Frankenstein) and requested service tickets for accounts with an SPN, then cracked them offline:

impacket-GetUserSPNs 'siemlab.local/Frankenstein:password123!' -dc-ip 10.0.30.10 -request

john krb_hash.txt --wordlist=mylist.txt

GetUserSPNs returned the password hash for the service accounts within the AD. I then used john and the custom mylist.txt file to crack the hash.

Stage 2 — DCSync

Using a Domain Admin account (reaper), impersonated a domain controller and replicated every credential hash out of the domain, including krbtgt:

impacket-secretsdump 'siemlab.local/reaper:Password123!@10.0.30.10' -just-dc -dc-ip 10.0.30.10

Dumping every hash from the entire AD allows me to move anywhere within the AD as an authenticated user and perform administrative actions.

Stage 3 — Pass-the-Hash (Lateral Movement)

Took the Administrator NTLM hash from the DCSync dump and used it to authenticate to the Windows 11 host, with no password, and get a shell:

impacket-wmiexec -hashes aad3b435b51404eeaad3b435b51404ee:<Administrator_NTLM> Administrator@10.0.30.40

Confirmed control on the target as siemlab\administrator on DESKTOP-JL7MAK5 (whoami, hostname). Moving to another machine with only a hash demonstrated lateral movement. I did not need a password to access an administrative shell.

3. DETECTION — SECURITY ONION (NETWORK)

Security Onion detected the lateral movement traffic from Kali (10.0.40.200). Key alerts:

ET INFO NTLMv1 Session Setup Response - Challenge (2067086)

The WMIExec alert captured the payload, showing the exact commands run over the pass-the-hash session.

Security Onion alerted on the NTLM session request, response, and authentication. It indicated that the protocol used was TCP on port 445. The alerts tell me that a client on the network is trying to negotiate an NTLM authentication session with a server. Security Onion flagged this as a potentially malicious or unusual authentication pattern across Windows operating systems. A huge piece of valuable information from SO was the alert for Impacket WMIExec Activity. This indicates that Impacket is a collection of Python classes aimed at interacting with network protocols, and WMIExec allows for executing commands on remote systems via Windows. The detection picked up on established server-bound flows containing specific binary patterns that are often associated with lateral movement or unauthorized access attempts. The four alerts above painted a clear picture of what happened in this case study.

On the host end Wazuh added the knowledge that an admin level session was created. The third ID, T1550.002, that Wazuh alerted to tells an entire story. Adversaries may “pass the hash” using stolen password hashes to move laterally within an environment, bypassing normal system access controls.

5. ANALYSIS — WHY THIS MATTERS

This attack chain used known TTPs by chaining techniques together to move from a low privilege user to compromising the entire domain. In this case it all happened very fast, the adversary went from standard user to having the ability to move anywhere within the domain and use administrative privileges in just a couple of minutes. Wazuh and SO need to be monitored regularly to catch an attacker's initial move within the network. The deeper they can dig in the harder it is to get them out. Network detection provided insight into the tools used and NTLM signatures, while host detection alerted to the logons, privilege assignment, and pass-the-hash technique. An attack chain like this is harder to catch because it may first appear as normal network activity or noise.

6. REMEDIATION & TAKEAWAY

To defend against this chain of attacks I would implement strong service account passwords or gMSA, which is when AD automatically manages the password for you. It is 240 characters long and is auto-rotated by Windows, and no person ever knows or types the password. I would limit Domain Admin accounts and replication rights to defend against DCSync. I would also enforce SMB signing to defend against pass-the-hash. I would be sure to watch Wazuh and Security Onion for Impacket signatures and unusual logon and privilege events.
