Detection & Configuration

The custom detection rules I wrote and validated, plus the intentional configuration work across the environment that makes those detections possible. Rules deploy to Wazuh at /var/ossec/etc/rules/local_rules.xml.

Custom Detection Rules

Rule 100150: Kerberoasting Detection

Validated

T1558.003  Fires on Event 4769 with RC4 encryption (0x17).

<group name="windows,kerberoasting,"> <rule id="100150" level="12"> <if_sid>60103</if_sid> <field name="win.system.eventID">^4769$</field> <field name="win.eventdata.ticketEncryptionType">^0x17$</field> <description>Possible Kerberoasting: RC4 service ticket requested (4769, 0x17)</description> <mitre><id>T1558.003</id></mitre> <options>no_full_log</options> </rule> </group>

Validated with:
impacket-GetUserSPNs siemlab.local/frankenstein:password123! -dc-ip 10.0.30.10 -request

Rule 100160: Noise Suppression

Validated

Silences benign DC01$ machine-account logon noise (level 0) to cut alert fatigue while keeping genuine logons alerting.

<group name="windows,noise_reduction,"> <rule id="100160" level="0"> <if_sid>60106</if_sid> <field name="win.eventdata.targetUserName">^DC01\$$</field> <description>Benign DC01 machine account logon - suppressed to reduce noise</description> </rule> </group>

Suppresses machine-account logon events that would otherwise flood the analyst queue.

Rule 100170: Auth From Attack Subnet

Validated

T1078  Fires on any authentication sourced from the attack VLAN (10.0.40.0/24).

<group name="windows,attack_subnet,"> <rule id="100170" level="12"> <if_sid>60103</if_sid> <field name="win.eventdata.ipAddress">^10\.0\.40\.</field> <description>Authentication from attack subnet (10.0.40.0/24) - possible intrusion</description> <mitre><id>T1078</id></mitre> </rule> </group>

Any successful or failed auth from the Kali attack subnet immediately raises a level-12 alert.

Configuration Edits

The environment configuration that generates and delivers the right telemetry to the SIEM: bridges, addressing, sensor interfaces, and Windows audit tuning.

Proxmox · Virtualization Host

SPAN Flood Script

Validated

A bash script that dynamically finds every port on the SPAN bridge (vmbr2) and forces it into flood mode with learning disabled, so all mirrored traffic reaches Security Onion.

#!/bin/bash # Apply SPAN flood settings to all ports on vmbr2 (SPAN bridge) for IFACE in $(bridge link | grep "master vmbr2" | awk -F: '{print $2}' | awk '{print $1}'); do bridge link set dev "$IFACE" flood on bridge link set dev "$IFACE" learning off bridge link set dev "$IFACE" mcast_flood on done

Bridge Architecture

Purpose-separated Linux bridges on the Proxmox host: vmbr1 for management (10.0.10.5/24), vmbr2 as the SPAN input from the mirror port, and vmbr3 as dedicated Security Onion management.

auto vmbr1 # management iface vmbr1 inet static address 10.0.10.5/24 gateway 10.0.10.1 auto vmbr2 # SPAN input from mirror port iface vmbr2 inet manual bridge-ports enp6s0f0 auto vmbr3 # dedicated SO mgmt iface vmbr3 inet manual bridge-ports enp6s0f1

A segmented static addressing scheme designed and configured across four VLANs, applied to every endpoint so the environment is predictable and fully documented.

Monitoring & Management Interfaces

The management interface (ens18) is set to a static IP on the SIEM VLAN, while the monitoring interface (ens19) is configured with autoconnect off so it passively sniffs mirrored traffic rather than carrying an address.

# ens18 - management address1=10.0.20.10/24 gateway=10.0.20.1 method=manual # ens19 - monitoring (passive) autoconnect=false method=auto
Active Directory · DC01

Audit Policy for Detection

Enabled the specific audit categories that generate the events the detection rules rely on, so attacks surface as security events. Key categories: Kerberos Service Ticket Operations, Logon/Logoff, Credential Validation, and Account Management.

Wazuh Agent Event-Channel Tuning

Curated which Windows events forward to the manager rather than shipping everything: a Security channel filtered to the detection-relevant Event IDs, a Sysmon channel, and a noise-exclusion query that drops high-volume events. This keeps the pipeline focused on what matters.

<query>Event/System[EventID=4662 or EventID=4625 or EventID=4624 or EventID=4768 or EventID=4769 or EventID=4776 or EventID=1102]</query>

Service Accounts as Attack Surface

Created domain service accounts with registered SPNs (MSSQL service principals) so the environment presents realistic, crackable service tickets for Kerberoasting to target.
